JT808 / JT1078 Test & Setup Guide

A practical manual for the testing team — from first build to live JT1078 stream verification.

Architecture

The repository contains two deployable components:

Fleet Simulator

Simulates any number of JT808 vehicle terminals using Netty event loops. Each terminal performs the full registration → authentication → heartbeat → location lifecycle and can emit real camera/mic streams (JT1078) or synthetic media. When DMS mode is enabled, a Python MediaPipe sidecar analyses the laptop webcam in real-time and reports eye-closure and distraction alarms via 0x0200 location reports.

Platform (jt808-platform)

The production-grade Spring Boot microservices stack: Reactor Netty gateway, Kafka event bus, ClickHouse telemetry persistence, alarm ingestion, and RTVS/CVNet media coordination. Decodes all standard JT808-2013 messages plus JT/T 1078-2016 video alarm TLVs (0x14–0x18) and the DMS TLV (0x65).

Quick Start — Embedded Server (Single Machine)

Uses the embedded ServerMain inside the simulator jar. No Kafka or ClickHouse needed. Good for protocol smoke tests.

  1. 1
    Start the embedded JT808 server 
    java -Djt808.transport=nio \
  -cp target/jt808-fleet-simulator-0.1.0-SNAPSHOT.jar \
  com.example.jt808sim.server.ServerMain \
  --config config/server.json
    Expected: JT808 alarm server listening on 0.0.0.0:7611
  2. 2
    Start the simulator (separate terminal) 
    java -Xms512m -Xmx1g \
  -XX:+UseG1GC \
  -Dio.netty.allocator.type=pooled \
  -Djt808.transport=nio \
  -jar target/jt808-fleet-simulator-0.1.0-SNAPSHOT.jar \
  --config config/fleet.json
    The dashboard appears and refreshes every second.
  3. 3
    Confirm authentication

    Within 30 seconds the dashboard should show Authenticated sessions = 10 (matching connectionCount in config/fleet.json).

ℹ

The embedded server does not have Kafka or ClickHouse. For full telemetry persistence, use the Platform Stack below.

Platform Stack — Infrastructure Setup

The platform services require Kafka, Redis, and optionally ClickHouse. Use Docker Compose for a local dev stack.

Minimal docker-compose.yml

# Save as docker-compose.yml in the repo root and run: docker compose up -d
services:
  kafka:
    image: confluentinc/cp-kafka:7.6.1
    ports: ["9092:9092"]
    environment:
      KAFKA_PROCESS_ROLES: broker,controller
      KAFKA_NODE_ID: 1
      KAFKA_CONTROLLER_QUORUM_VOTERS: "1@kafka:9093"
      KAFKA_LISTENERS: PLAINTEXT://0.0.0.0:9092,CONTROLLER://0.0.0.0:9093
      KAFKA_ADVERTISED_LISTENERS: PLAINTEXT://localhost:9092
      KAFKA_CONTROLLER_LISTENER_NAMES: CONTROLLER
      KAFKA_LOG_DIRS: /tmp/kraft-logs
      CLUSTER_ID: MkU3OEVBNTcwNTJENDM2Qg

  redis:
    image: redis:7-alpine
    ports: ["6379:6379"]
    command: redis-server --save "" --appendonly no

  clickhouse:
    image: clickhouse/clickhouse-server:24.3
    ports: ["8123:8123", "9000:9000"]
    ulimits:
      nofile: { soft: 262144, hard: 262144 }
docker compose up -d
# Verify all three containers are healthy
docker compose ps

Each service has an application.yml with all settings defaulting to 127.0.0.1. Enable Kafka and Redis with the flags below. Override on the command line with --jt808.gateway.kafka.enabled=true or by editing application.yml.

gateway-service

      enabled: true                   # set to true when Redis is running

Test Scenarios

Run scenarios in order. Each builds on the previous. Wait for the stack to reach steady state (~60 s) before checking pass criteria.

T-01 Connection & Authentication

What it tests: Full JT808 lifecycle — register (0x0100), get response (0x8100), authenticate (0x0102), receive ack (0x8001).

Wait: 30 s after simulator starts.

T-02 GPS Telemetry Pipeline

What it tests: Terminals send periodic 0x0200 location reports → gateway publishes to Kafka telemetry.gps → telemetry-service writes to ClickHouse vehicle_gps.

Wait: 2 minutes after T-01 passes.

# Row count should grow with each run
curl -s "http://localhost:8123/?query=SELECT+count()+FROM+jt808.vehicle_gps"

# Latest GPS rows
curl -s "http://localhost:8123/?query=SELECT+device_id,latitude,longitude,gps_time+FROM+jt808.vehicle_gps+ORDER+BY+gps_time+DESC+LIMIT+5+FORMAT+PrettyCompact"

PASS when row count climbs and coordinates change between rows.

T-03 Alarm Ingestion

What it tests: Simulator triggers alarms via the AlarmEngine → gateway detects bit transitions in 0x0200 alarm word → publishes AlarmEvent to Kafka → alarm-service writes to vehicle_alarm.

Trigger an alarm: The simulator's AlarmEngine fires simulated alarms periodically. Alternatively, observe naturally over 5 minutes (overspeed and fatigue alarms cycle automatically).

curl -s "http://localhost:8123/?query=SELECT+device_id,alarm_type,alarm_name,cleared,gps_time+FROM+jt808.vehicle_alarm+ORDER+BY+gps_time+DESC+LIMIT+10+FORMAT+PrettyCompact"

For video alarms, verify the extended columns:

curl -s "http://localhost:8123/?query=SELECT+video_alarm,video_signal_lost_channels,fatigue_degree+FROM+jt808.vehicle_alarm+WHERE+video_alarm+>+0+LIMIT+5+FORMAT+PrettyCompact"

PASS when at least one alarm row exists with cleared=0 and a matching cleared=1 row confirming the lifecycle.

T-05 RTVS Gateway — Live Video Command

What it tests: RTVS calls /VideoControl → gateway sends 0x9101 to terminal → terminal opens JT1078 session → gateway writes AVParameters to Redis.

Steps:

  1. Confirm AVParameters key was written after terminal auth (from 0x1003 response): 
    redis-cli GET "AVParameters:00000000000000000001"
    Expected: JSON with VideoEncoding, MaxVideoChannels, etc.
  2. Send a live-start command via the RTVS gateway API: 
    curl -G "http://localhost:8888/VideoControl" \
  --data-urlencode "Content=<JT808-hex-for-0x9101>"
    Expected response: 1 (accepted) or a command ID.
  3. Verify OCX_ORDERINFO_<commandId> appears in Redis: 
    redis-cli KEYS "OCX_ORDERINFO_*"

PASS when command is accepted, AVParameters key exists, OCX_ORDERINFO key written.

T-08 DMS — Driver Monitoring via Webcam

What it tests: Real camera capture via FFmpeg → H.264 live stream to RTVS (simultaneous with DMS analysis), Python MediaPipe sidecar detecting eye closure and distraction → 0x65 DMS TLV embedded in 0x0200 location reports → platform decodes and writes dms_alarm_type / dms_fatigue_degree / dms_alarm_flags to ClickHouse.

Prerequisites: Webcam attached, FFmpeg installed, Python DMS sidecar running (step 7 above), simulator started with config/camera-host.json and "dms": {"enabled": true}.

  1. 1
    Start DMS sidecar 
    python dms-sidecar/dms_server.py
curl http://localhost:7500/health   # → {"ok":true}
curl http://localhost:7500/dms/state  # should show faceDetected, alarmFlags
  2. 2
    Start simulator with camera config 
    java -jar target/jt808-fleet-simulator-0.1.0-SNAPSHOT.jar \
  --config config/camera-host.json
    Watch gateway log for: started ffmpeg camera capture backend
  3. 3
    Trigger fatigue alarm — close eyes for 3+ seconds

    Close your eyes and hold for 3 seconds while facing the webcam. The sidecar calculates Eye Aspect Ratio (EAR) below 0.25 threshold and increments the fatigue degree. After ~3 s the degree reaches 7+, FLAG_FATIGUE is set in alarmFlags, and the next 0x0200 will contain the 0x65 TLV.

    # Watch the sidecar state in real time
watch -n 0.5 'curl -s http://localhost:7500/dms/state | python3 -m json.tool'
    Expected when eyes closed: "fatigueDegree": 7, "alarmFlags": 1, "primaryAlarm": 1
  4. 4
    Trigger distraction alarm — look away from camera

    Turn your head away so MediaPipe loses the face. Within two seconds the sidecar sets alarmFlags: 2 (FLAG_DISTRACTION) and primaryAlarm: 2.

  5. 5
    Trigger seatbelt alarm 
    curl -X POST http://localhost:7500/dms/seatbelt \
  -H "Content-Type: application/json" -d '{"worn":false}'
    Expected sidecar state: "alarmFlags": 16, "primaryAlarm": 5 (FLAG_NO_SEATBELT = bit 4 = 16)
  6. 6
    Verify in ClickHouse 
    curl -s "http://localhost:8123/?query=SELECT+device_id,dms_alarm_type,dms_fatigue_degree,dms_alarm_flags,gps_time+FROM+jt808.vehicle_alarm+WHERE+dms_alarm_flags+>+0+ORDER+BY+gps_time+DESC+LIMIT+10+FORMAT+PrettyCompact"

Troubleshooting

Simulator connects but never authenticates

  • Check jt808.gateway.auth-code in gateway config matches the auth token the simulator expects.
  • Check gateway logs for decode errors on 0x0100 / 0x0102 messages.
  • Confirm gateway is running: curl localhost:8888/actuator/health

Media sessions never appear

  • Verify jt1078.host in fleet.json points to the machine running RTVS (not 127.0.0.1 on a two-machine setup).
  • Check RTVS ingest port is open: nc -zv SERVER_IP 1078
  • Confirm mediaCapableTerminalCount ≥ 1 in fleet config.

ClickHouse tables empty

  • Confirm telemetry-service and alarm-service started with kafka.enabled=true.
  • Check Kafka topic has messages: kafka-console-consumer.sh --bootstrap-server localhost:9092 --topic telemetry.gps --max-messages 3
  • Check ClickHouse is up: curl http://localhost:8123/ping → should return Ok.

Invalid checksum count > 0

✕

Stop the test immediately. Capture a packet trace and file a bug report: sudo tcpdump -i any -w /tmp/capture.pcap port 7611

Port already in use

ss -tlnp | grep -E '7611|7612|1078|8888|8090'

Out of file descriptors (simulator or gateway)

ulimit -n 200000               # raise limit in the shell running the process
cat /proc/$(pgrep java)/limits  # check effective limit of running JVM

DMS sidecar not detecting face

  • Check curl http://localhost:7500/dms/state — if alarmFlags: 32 (FLAG_CAM_BLOCKED), the camera index is wrong. Set DMS_CAMERA_INDEX=1 (or 2) and restart.
  • Ensure sufficient lighting — MediaPipe Face Mesh needs a clear frontal view.
  • If running inside a VM or remote desktop, V4L2 camera forwarding may be needed.

DMS alarms not appearing in ClickHouse

  • Confirm "dms": {"enabled": true} in the fleet config and the sidecar is running before the simulator starts.
  • Check sidecar state: curl http://localhost:7500/dms/state — alarmFlags must be non-zero before any 0x65 TLV is sent.
  • Verify the gateway is decoding 0x65: look for dmsAlarmType in gateway debug logs (-Dlogging.level.com.example=DEBUG).
  • Confirm platform AlarmEvent.dmsAlarmFlags flows through to alarm-service — check Kafka topic telemetry.alarm.

FFmpeg camera stream fails to start

  • Check the device exists: ls -l /dev/video*
  • Test FFmpeg directly: ffmpeg -f v4l2 -i /dev/video0 -vframes 1 test.jpg
  • If another process holds the device, find it: fuser /dev/video0
  • On macOS, change capture flags to -f avfoundation in FfmpegCameraFrameSource.buildCommand().

Redis session key missing

  • Confirm redis.enabled=true in gateway config.
  • Confirm Redis is running: redis-cli ping → PONG
  • Keys are set after the terminal sends 0x0100 and 0x0102 — check the gateway log for those message IDs.
